I. Problem Definition & Domain Research

The "Why" — Real-World Problem

Independent musicians, hobbyists, and content creators frequently compose a primary melody — hummed, played on a keyboard, or sketched as a MIDI file — but lack the musical theory expertise or financial resources to hire session musicians. Professional arrangers charge hundreds of dollars per session, and existing software tools such as GarageBand still demand substantial music-theory knowledge to produce genre-coherent multi-instrument arrangements.

MuseFlow addresses this gap directly: given any melody (including a hummed voice recording), the system automatically generates a complete, genre-consistent multi-instrument band arrangement in Classical, Lofi, or Rock style, with no music-theory requirement from the user. The secondary problem solved is the "blank canvas" problem in music production — the paralysis creators feel when starting from silence. MuseFlow acts as an AI co-pilot, providing a full arrangement starting point from the simplest possible input.

Data Provenance

Training data is sourced from two publicly available, royalty-free corpora:

  • Lakh MIDI Dataset (LMD): Approximately 45,000 MIDI files covering diverse genres. Files were automatically classified into Rock and Lofi categories using keyword heuristics on filenames and MIDI program numbers (e.g., distorted guitar → Rock; electric piano/vibraphone → Lofi). Up to 4,000 files per genre were selected for training.
  • music21 Bach Chorales (MIT Corpus): A curated set of J.S. Bach four-part chorale harmonisations obtained via the music21 library's built-in corpus. These form the Classical genre subset and are ideal because they already exhibit clean Melody + Band track separation.

Missing Values & Potential Biases

Identified Data Limitations
  • Single-track Lofi files: Many Lofi-tagged LMD files contained only one instrument track, making Seq2Seq pair creation impossible (which requires ≥2 tracks). A custom rescue.py script was written to synthetically split single-track files into melody and accompaniment.
  • Genre labelling noise: LMD carries no official genre tags; keyword-based classification may misclassify edge cases (e.g., a Rock ballad labelled as Lofi).
  • Western music bias: Both datasets are overwhelmingly Western tonal music. The model will underperform on non-Western scales, microtonal music, or non-standard rhythmic structures.
  • Melody extraction heuristic: The melody track is identified as the track with the highest average MIDI pitch — a reasonable but imperfect heuristic that can occasionally select a high-register accompaniment part instead of the true melodic line.

II. Methodology & Design Choices

Preprocessing Logic

Raw MIDI files cannot be fed directly into a neural network. The preprocessing pipeline converts them into integer token sequences through the following steps:

  1. Loading & Validation: Each file is loaded with symusic.Score. Files with fewer than 2 tracks are skipped immediately, as they cannot provide a melody/band split.
  2. Melody Extraction: The track with the highest average pitch is designated the prompt (melody); all remaining tracks form the target (band).
  3. REMI Tokenisation: Both the melody and band scores are independently tokenised using MidiTok's REMI scheme with 32 velocity bins, chord annotations, program numbers, and tempo events enabled.
  4. Genre Token Injection: The genre control token (10000/10001/10002) is prepended to the band token sequence — this is the core conditioning mechanism.
  5. Padding & Truncation: Prompt sequences are padded or truncated to exactly 128 tokens; target sequences to 384 tokens. Padding uses integer 0, which is ignored during loss computation via CrossEntropyLoss(ignore_index=0).
  6. Inference-time Cleaning: Uploaded audio or MIDI is cleaned via clean_melody_score(), which selects the busiest track, filters notes shorter than 40 ticks as noise, and snaps all onsets and durations to an 8th-note grid.
Why use integer 0 for padding instead of a dedicated [PAD] token?

REMI token ID 0 corresponds to a null/empty event that never appears in real music data. The ignore_index=0 flag in CrossEntropyLoss ensures padded positions contribute zero gradient, equivalent to a masked loss — without the overhead of a separate vocabulary entry.

Architecture Selection — Why a Transformer?

The core task is sequence-to-sequence music generation: given a melody token sequence, generate a coherent band accompaniment. Three model families were evaluated:

ArchitectureDecisionJustification
LSTM / GRU (RNN)RejectedPoor long-range dependency modelling. At 512 tokens, RNNs suffer from vanishing gradients and cannot attend back to the beginning of a melody phrase.
CNN (WaveNet-style)RejectedBest suited for raw audio waveforms, not discrete symbolic token sequences. Very deep stacks are needed to achieve a 512-token receptive field.
Decoder-only TransformerSelectedSelf-attention directly models all pairwise token relationships regardless of distance. The causal mask enforces autoregressive generation. Genre tokens embed naturally as control signals.

The final architecture is a 4-layer, 8-head causal Transformer with the following hyperparameters:

Sampling strategy: At inference time, token selection uses temperature scaling (T = 0.8–0.9), top-k filtering (k = 50), and nucleus (top-p) sampling (p = 0.95). This combination prevents the model from always selecting the highest-probability token (producing repetitive output) while avoiding very low-probability nonsense tokens.

III. Failure Analysis

Rather than presenting only the final successful configuration, this section documents two significant failures encountered during development and the lessons each revealed.

Failure 1 — Deeper Network (6 Layers) Caused Immediate Overfitting

Our initial hypothesis was that more Transformer layers would produce more musically coherent arrangements. A 6-layer version of MuseFlowTransformer was trained on the same dataset under identical conditions.

What happened:

By epoch 5, training loss had dropped to approximately 0.72, while held-out validation perplexity began rising. By epoch 12, the model reproduced near-exact fragments from training files — a clear sign of memorisation rather than generalisation. Generated output sounded plausible in isolation but was effectively identical across different melody inputs, meaning genre conditioning had collapsed.

Root cause:

With only approximately 3,000 usable training pairs, a 6-layer model (~15M parameters) had enough capacity to memorise training sequences rather than learn the underlying melody-to-band mapping. The data-to-parameter ratio was too low.

Fix applied:

The model was reduced to 4 layers (~10M parameters). L2 regularisation was added via weight_decay = 0.01 in AdamW, and the dataset was scaled to 2,000 files per genre to improve the data-to-parameter ratio.

Failure 2 — Phase 1 Single-Sequence Training Produced Incoherent Genre Mixing

The first training approach concatenated all tracks of a MIDI file into one long token sequence and trained the model to predict the next token autoregressively. Genre tags were appended at the end of the entire sequence.

What happened:

Generated output showed no meaningful genre differentiation — requesting "Rock" would often produce output nearly identical to "Classical". The model learned global musical patterns but established no causal chain between the genre token and specific instrument choices.

Root cause:

In a causal (left-to-right) model, when the genre token appears at the end of a very long sequence, the model has already committed to most of its output distribution before "seeing" the genre tag. The tag arrives too late to meaningfully influence generation.

Fix applied:

The entire pipeline was redesigned into a Seq2Seq format. The genre token is now prepended at the start of the target (band) sequence — position 129 in the full 512-token context. The model now learns to condition all subsequent band tokens on the genre tag from the very first generated token onward.

Training Dynamics — Final Model (v4)

The final model was trained for 50 epochs using AdamW (lr = 3×10⁻⁴, weight_decay = 0.01) with Cosine Annealing LR scheduling. Mixed FP16 precision was used on GPU to halve VRAM consumption. Gradient clipping at max_norm = 1.0 was applied to prevent exploding gradients in early epochs.

Post-Processing Pipeline

Raw Transformer output is passed through several post-processing stages before being saved as a MIDI file to ensure rhythmic accuracy and cohesive playback:

  1. Track Consolidation: The REMI decoder often fragments a single instrument into multiple 1-note tracks. The pipeline groups and merges all tracks sharing the same MIDI program number into unified instrument tracks.
  2. Aggressive Vertical Alignment: To prevent the band from "waiting" for the input melody to finish, all generated instrument tracks are force-shifted so their first note starts at exactly tick 0 alongside the main melody.
  3. Strict Melody Window (Duration Clipping): To prevent accompaniment from trailing off, the exact duration of the user's melody is calculated, and all secondary instruments are strictly clipped so they stop the moment the melody ends.
  4. Grid Quantisation (16th notes): Snaps all note onsets and durations to a 16th-note grid (grid_division = 4), eliminating rhythmic drift caused by model uncertainty at fine-grained time positions.
  5. Velocity Normalisation: Velocity is remapped using 60 + (v × 0.4), compressing the dynamic range to avoid jarring accent spikes while preserving relative dynamics.
  6. Duration Humaniser (±5% jitter): After quantisation, a small random duration jitter of ±5% is reapplied to prevent the output from sounding mechanically rigid, restoring the feel of a human performance.

V. Ethical Reflection & Limitations

Where This Model Would Fail in the Real World

  • Non-Western music: The model is trained exclusively on Western tonal MIDI data. It will produce incorrect or culturally inappropriate arrangements for Arabic maqam scales, Indian ragas, or pentatonic-only melodies by forcing them into Western harmonic structures.
  • Atonal or experimental inputs: Melodies not conforming to 12-tone equal temperament will be snapped to the nearest tempered pitch during preprocessing, destroying intentional dissonance or microtonality.
  • Very short melodies (fewer than 8 notes): The melody looping mechanism tiles short inputs. If the source melody lacks sufficient harmonic movement, looping produces repetitive, unmusical accompaniments.
  • Live performance tempo variation (rubato): Significant tempo flexibility in hummed input is transcribed by Basic Pitch with fixed tempo assumptions, causing rhythmic misalignment between the transcribed MIDI and the model's grid-based training distribution.
  • Copyright edge cases: If a user hums a copyrighted melody (e.g., a well-known pop hook), the generated arrangement combined with the input may constitute a derivative work, raising intellectual property concerns — even though the AI generated the accompaniment independently.

Ethical Implications of Deployment

1. Training Data Copyright

The Lakh MIDI Dataset was assembled by scraping publicly indexed MIDI files from the internet. While LMD is widely used in academic research, the original files may be transcriptions of copyrighted songs. Training on these files without explicit licensing sits in a legally grey area under current AI copyright law, as model weights encode statistical patterns derived from these works.

2. Musician Displacement

MuseFlow can produce in seconds what a session musician arranger would take hours to compose and record. Deploying such a system at scale risks devaluing the work of professional arrangers, session musicians, and composers — particularly those in entry-level or freelance roles. This is not a hypothetical concern: AI music generation tools are already being used to replace stock music composers on commercial licensing platforms.

3. Cultural Misrepresentation

The genre tags (Classical, Lofi, Rock) are broad, Western-centric categories. Labelling output as "Classical" carries associations with European art music tradition. Users from different cultural backgrounds may incorrectly assume the output reflects their own musical tradition, and the model's Western bias could reinforce a narrow global view of what "classical" or "traditional" music means.

4. Homogenisation of Creative Output

While democratising music production is a genuine social benefit, AI-generated arrangements flooding creative platforms risk producing homogenised output that all shares the statistical fingerprint of the training data. Over time, this could reduce musical diversity on streaming and content-creation platforms.
